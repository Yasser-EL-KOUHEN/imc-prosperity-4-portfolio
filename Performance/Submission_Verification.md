> Each round's submitted `.py` file lives in `performance/algorithmic trading/round N/{submission_id}.py`. This is the canonical "what we ran in the competition" — it differs from some of the vault's repository traders. This page is the provenance ledger.

## Critical discrepancies

### R1: v2 was submitted, NOT v3

The R1 vault `round1/trader.py` is documented as v3 ("FV anchor blend") and labelled in vault pages as "the R1 trader". **The actually-submitted R1 trader was v2 ("book-anchored passive quotes")** — confirmed by docstring on line 2 of `272592.py`:

```
Prosperity 4 -- Round 1 Trader  (v2 -- book-anchored passive quotes)
```

vs vault's:
```
Prosperity 4 -- Round 1 Trader  (v3 -- FV anchor blend)
```

**Implication.** The realized R1 algo PnL of **$98,172** was generated by **v2**, not v3. v3 (with FV anchor blend FV = 0.85·EMA + 0.15·10000 and ρ-bucketed inventory tweaks) was developed locally — likely after the R1 submission deadline — but was never submitted to R1. v3 *was* submitted to R2 (since `round2/trader.py` matches v3 + MAF bid).

**Performance check.** v3 was selected via `aco_opt_sweep.py` for being +4.1% on ACO local (51,040 → 53,116). That improvement was on local backtest only and was never tested on real engine in R1.

**Vault correction needed:** any page claiming "R1 algo was v3" should be amended. The realized R1 algo was v2.

### R3: vault has TWO traders, neither matches the submission — but the timeline explains it

The R3 vault has both `round3/trader.py` (43,435 bytes, "v3") and `round3/trader_final.py` (40,334 bytes, also "v3"). The submitted file `486282.py` is **36,290 bytes**. Three "v3" files; three different hashes.

**The clean timeline (user-clarified):**

| File | Role | When |
|---|---|---|
| `round3/trader.py` (991 lines) | dev/research version with extra scaffolding (4 box_state functions removed downstream) | pre-submission |
| `486282.py` (880 lines, smallest) | **actually submitted to R3** — generated the $40,800 algo PnL | submission |
| `round3/trader_final.py` (940 lines) | **40% improved post-R3 refinement** — Phase 4 ρ-sweep tuning + Phase 10 TTE recalibration applied. **Used as the R4 basis**, not submitted to R3 | post-R3, pre-R4 |

**Parameter differences (`486282.py` vs `trader_final.py`):**

| Parameter | R3 submitted (`486282.py`) | trader_final (R4 basis) | Source |
|---|---|---|---|
| `small_rho` | 0.12 | 0.08 | Phase 4 ρ sweep refinement (post-R3) |
| `large_rho` | 0.48 | 0.42 | Phase 4 ρ sweep refinement (post-R3) |
| `initial_tte_days` | 5.0 | 7.0 | Phase 10 Addendum 2 TTE recal (post-R3) |
| Other VEV params | various | refined | per-strike σ recalibration |

**Implication for vault analytics:**
- The realized R3 algo PnL of **$40,800 was generated with the un-refined parameters** (small_rho=0.12, large_rho=0.48, TTE=5.0), not the Phase-4-winner values (0.08, 0.42, 7.0) documented in vault parameter pages.
- The Phase 4 ρ sweep and Phase 10 TTE recalibration are real and valid — they describe **post-R3 refinements that informed R4**, not the parameters that produced R3's leaderboard result.
- The R4 algo PnL of $57,048 *did* use the refined Phase 4 + Phase 10 parameters (since `trader_final.py` is the R4 basis, and R4 vault `trader.py` exactly matches the submitted `544306.py`).
- HYDROGEL real-engine performance was nearly identical R3 vs R4 (+55,100 vs +56,200, 1.99% variance). The parameter refinements between submissions contributed marginal lift; HYDROGEL's underlying alpha was robust to either parameter set.

**Vault correction needed:** Phase 4 / Phase 10 / Phase 11 backtest pages should annotate that the documented parameter winners (small_rho=0.08, large_rho=0.42, TTE=7.0) were the *post-R3 refinement* values that informed R4 — the R3 leaderboard result came from earlier values. The phase backtests are legitimate research artifacts; they just need their submission attribution clarified.

### R4: exact match — fully verified

`round4/trader.py` and `544306.py` are byte-identical. The R4 algo PnL of $57,048 was generated by exactly the file in the vault. No correction needed.

### R5: v42 submitted, 1-byte newline diff only

The submitted `581865.py` differs from `round5/strategies/round5_v42_trader.py` by exactly one trailing newline. Substantively identical. v42 (as documented in vault) **was** submitted. The realized R5 algo PnL of $57,911 was generated by v42.

### R2: v3 + MAF submitted, 1-byte newline diff only

Same as R5 — substantively identical, just newline. Vault's `round2/trader.py` *was* the submission.

## Implications for vault accuracy

The R4 and R5 vault traders are exact (or effectively exact). R2 is also fine. **R1 and R3 vault traders are misleading**:

- R1: vault advertises v3 as "the trader", but realized PnL came from v2. Pages that derive lessons or parameter values from "what R1 ran" need to be re-checked against `272592.py` (v2), not `round1/trader.py` (v3).
- R3: vault has two candidate "v3" files, the submitted is a third. Lessons derived from `round3/trader.py` or `trader_final.py` parameter values cannot be confidently attributed to the realized $40,800 algo PnL.

## What changed at the realized-PnL level

For R1: realized $98,172 came from v2's logic:
- ACO: anchor to book — passive_bid = best_bid + 1, passive_ask = best_ask − 1 (always 1 tick inside bot's spread, guaranteed fills)
- IPR: aggressive-only — take ALL available asks immediately, build max-long position (80 units), then hold; post passive buy at best_bid + 1; never post passive sells

For R3: realized $40,800 came from `486282.py`. We have not yet diffed the three v3 candidates to identify what differs. **Action item:** detailed diff to attribute the realized PnL components to specific code paths.

## What's in the .log files

No additional sandbox stdout / order-trace info beyond what the JSONs provide. The JSONs are the canonical analysis source; the .log files exist as raw exports.

## Provenance recommendations (carry-forward to Prosperity 5)

1. **Always commit the actually-submitted file with its submission ID into the repo.** The fact that `performance/` contains the submitted code is what makes this verification possible. Without it, we'd be guessing.
2. **Don't develop multiple "v3" variants without renaming.** The R3 trader.py / trader_final.py / submitted-486282 confusion would have been avoided with strict semantic versioning (v3.0 / v3.1 / v3-submitted).
3. **Tag the git commit at submission time.** A `submitted-r3` tag pointing to the exact commit would have removed all R3 ambiguity.
